# IssueOps Model

PolarSwarm treats GitHub Issues and Comments as the collaboration protocol.

IssueOps in this project means using Issues, Pull Requests, Comments, Labels, and lightweight Actions as the visible coordination surface. It is not itself a GitHub platform guarantee; platform-specific assumptions are covered by `github-capabilities.md`.

## Work Units

Only user-visible work is represented as Issues. Internal retries, temporary execution steps, fallback attempts, and local diagnostics are recorded locally or as structured comments, not as new Issues by default.

Recommended product issue categories are:

| Type | Primary role | Required content |
|---|---|---|
| Epic | `orchestrator` / `architect` | Goal, acceptance criteria, related milestones, and child work summary. |
| Feature | `developer` | User story, acceptance criteria, priority, and expected deliverables. |
| Bug | `debugger` -> `developer` | Steps to reproduce, expected behavior, actual behavior, and environment. |
| Task | Any role | Description, deliverables, owner role, and done criteria. |
| Spike | `architect` | Question, time box, research scope, and done criteria. |
| Security | `security` | Severity, affected components, evidence, and expected remediation. |
| Compliance | `compliance` | Standard, level, audit scope, deadline, and required evidence. |
| Test | `tester` | Test type, scope, success criteria, and linked feature or bug. |
| Docs | `documenter` | Audience, doc type, linked code or feature, and expected format. |
| Release | `devops` / `orchestrator` | Version, milestone, release checklist, and release note source. |

Issue Types are optional GitHub-native structure. When unavailable, equivalent `type:*` Labels and templates provide the degraded representation.

## Label Projection

Labels are a human-readable projection of state, not the source of truth. The source of truth is the replayable GitHub event stream plus versioned policy.

### Status Labels

Exactly one `status:*` label should be present at a time:

- `status:pending-triage`
- `status:new`
- `status:triaged`
- `status:assigned`
- `status:in-progress`
- `status:blocked`
- `status:review`
- `status:rework`
- `status:done`
- `status:abandoned`

The orchestrator enforces mutual exclusion by removing old `status:*` labels before adding the new one.

State transitions should follow this shape:

```text
pending-triage -> new -> triaged -> assigned -> in-progress -> review -> done
                                             \-> blocked -> assigned
                                             \-> rework -> in-progress -> review
                                             \-> abandoned
```

External or unauthorized submissions start at `status:pending-triage`. Authorized work can start at `status:new`.

### Agent Labels

Agent labels show the current responsible role:

- `agent:orchestrator`
- `agent:architect`
- `agent:developer`
- `agent:reviewer`
- `agent:security`
- `agent:documenter`
- `agent:debugger`
- `agent:merger`
- `agent:tester`
- `agent:devops`
- `agent:compliance`
- `agent:human`

Agent labels use roles, not execution backends.

### Decision Labels

Decision Labels record the current projected decision state and may coexist with `status:*`:

- `decision:approved`
- `decision:rejected`
- `decision:needs-info`

The orchestrator must clear stale `decision:*` Labels before starting a new decision cycle, so old approvals do not appear to apply to later work.

### Priority and Effort Labels

Priority Labels:

- `priority:critical`
- `priority:high`
- `priority:normal`
- `priority:low`

Effort Labels:

- `effort:1`
- `effort:2`
- `effort:3`
- `effort:5`
- `effort:8`
- `effort:13`

## Comments

All machine-readable comments use fenced code blocks. The first line must remain human readable.

Important comment types:

- `polarswarm-msg`: routeable agent command envelope.
- `polarswarm-agent-result`: agent stage result.
- `polarswarm-checkpoint`: human checkpoint request.
- `polarswarm-checkpoint-response`: human or autonomous checkpoint response.
- `polarswarm-review-result`: reviewer, security, or compliance review result.
- `polarswarm-rework-request`: structured rework request.
- `polarswarm-rework-response`: structured rework completion.
- `polarswarm-test-result`: test result.
- `polarswarm-merge-result`: merge result and conflict analysis.

Routeable work commands use `polarswarm-msg`. Audit-only records use their dedicated fenced block and must not be interpreted as routeable commands unless paired with a valid `polarswarm-msg`.

Required envelope fields for routeable messages:

| Field | Purpose |
|---|---|
| `version` | Message schema version. |
| `msg_id` | Unique message ID generated by the sender. |
| `correlation_id` | Root workflow or task correlation ID. |
| `causation_id` | Parent message ID, or `null` for root messages. |
| `from` | Sender role or system actor. |
| `to` | Target role or system actor. |
| `type` | Routeable message type. |
| `issue_ref` | GitHub Issue number. |
| `payload` | Type-specific command data. |
| `ts` | Sender timestamp. |

The local runtime must record processed `msg_id` values and skip duplicates. Edited comments may refresh display state, but must not replay workflow transitions by themselves.

## Message Types

Core routeable message types:

| Type | Meaning |
|---|---|
| `task_assign` | Orchestrator assigns work to a role. |
| `task_start` | Agent reports work has started. |
| `task_done` | Agent reports completion and moves work toward review. |
| `task_handoff` | One role hands work to another role. |
| `blocked_notify` | Agent reports a dependency or external blocker. |
| `unblocked_notify` | Orchestrator resumes blocked work. |
| `request_review` | Agent asks for review or human confirmation. |
| `escalate` | Agent cannot safely proceed and requests intervention. |
| `checkpoint_approve` | Human or autonomous policy approves a checkpoint. |
| `complete_approve` | Reviewer or human accepts completed work. |
| `reject` | Work is rejected or abandoned. |
| `review_result` | Reviewer, security, or compliance result. |
| `rework_request` | Reviewer or tester asks for changes. |
| `rework_done` | Implementer reports rework completion. |
| `test_result` | Tester posts structured test result. |
| `merge_request` | Orchestrator asks merger role to analyze or resolve a merge. |
| `merge_result` | Merger posts structured merge result. |
| `triage_approve` | Authorized actor promotes `pending-triage` work. |
| `triage_reject` | Authorized actor rejects `pending-triage` work. |

`review_result=changes_requested` must produce both a rework audit record and a routeable `rework_request`. `review_result=rejected` is reserved for unacceptable design, severe risk, or policy failure and should normally escalate before abandonment.

## Pull Requests

Pull Requests must link back to the driving Issue and include:

- Summary
- Linked Issue
- Agent Worklog
- Verification
- Risk

PRs are not considered complete unless the required agent result and review comments exist.

## Human Checkpoints

Human checkpoints are durable GitHub events first and local UI prompts second. When a checkpoint triggers, the orchestrator must:

1. Write a `polarswarm-checkpoint` Comment.
2. Project the Issue with `agent:human` and an appropriate pending decision state.
3. Show the local TUI prompt based on the already-written checkpoint.
4. Write a `polarswarm-checkpoint-response` Comment after the user responds.
5. Resume reconciliation from GitHub state.

If the local process restarts between steps, the unresolved GitHub checkpoint remains the recovery source.

## Security Gate

External Issues and Comments must not automatically spend agent resources.

- Authorized actors are defined by `.polarswarm/access.toml`.
- Non-authorized Issue submissions enter `status:pending-triage` and `agent:human`.
- Authorized users can use `/triage` to promote the Issue or `/reject` to close it.
- Blacklisted users can be automatically closed by the triage workflow.
- Machine messages may require HMAC signing when enabled.

GitHub does not provide Label-level ACL. Unauthorized Label changes are detected and corrected by reconciliation and, when installed, the Label guard workflow.

## State Authority

The authoritative workflow history is the replayable GitHub Issue, PR, Comment, Label, and close-state event stream plus versioned policy. Local SQLite state is a cache for cursors, leases, idempotency, metrics, and capability information.

When projected Labels conflict with replayed history, the orchestrator should repair Labels to match the replayed state instead of trusting the visible projection.
